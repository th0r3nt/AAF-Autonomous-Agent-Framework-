# 🌌 Autonomous Agent Framework

**AAF (Autonomous Agent Framework)** - полноценная OS-level сущность, асинхронный ИИ-агент и системный администратор, построенный на микросервисной архитектуре.

Большинство современных Open-Source агентов страдают от трех проблем: амнезия (забывают контекст через 5 шагов), зацикливание (бесконечные ReAct-петли) и ад зависимостей при установке. **AAF решает их все.**



## 🔥 Киллер-фичи (Почему AAF работает иначе?)

  
*   **🧠 Тройная гибридная память (State-of-the-Art):** система не теряет контекст. Она комбинирует Vector DB (семантика и поток сознания), PostgreSQL (жесткие правила и долгосрочные задачи) и GraphRAG на базе KuzuDB (нелинейные ассоциации и интуиция агента).

*   **⚡ Асинхронная шина событий (EventBus):** мозг агента полностью отвязан от сенсоров. Сообщения из Telegram, системные логи и фоновые задачи падают в единую очередь приоритетов (PriorityQueue). Агент сам решает, на что реагировать прямо сейчас, а что оставить на потом.

*   **🐝 Swarm System & Docker Sandbox:** агент не выполняет тяжелую рутину сам. Он динамически спавнит "одноразовые" Docker-контейнеры (Workers) для парсинга, анализа логов или выполнения Python-скриптов в абсолютно изолированной песочнице.

*   **🚀 Параллельный ReAct Engine:** в отличие от стандартных фреймворков, AAF вызывает инструменты параллельно. Нужно узнать погоду, прочитать файл и загуглить новость? Она сделает это одновременно за один шаг (через `asyncio.gather`).

*   **🎭 Zero-Code кастомизация личности:** чтобы полностью изменить характер агента, не нужно знать Python. Достаточно отредактировать 3 обычных текстовых файла в папке `config/personality`.

*   **💸 Бесплатная работа (API Rotator):** встроенный менеджер ключей использует алгоритм Round-Robin. Если один ключ Gemini ловит лимит (429), система быстро переключается на следующий.
  
## 🏗 Архитектура системы

Архитектура построена на строгом разделении слоев (Layers 0-4). Все модули общаются исключительно через асинхронный EventBus.

  
## ⚠️ Важное предупреждение о безопасности (Docker-in-Docker)

Архитектура AAF использует проброс сокета Docker (`/var/run/docker.sock`) внутрь контейнера `agent_core`. Это необходимо для того, чтобы агент мог динамически создавать изолированные песочницы (контейнеры-воркеры) для выполнения Python-скриптов и анализа кода.

**Что это значит для вас:** Агент имеет права администратора (root) над Docker-демоном вашей хост-машины. Не запускайте проект на production-серверах с критически важными данными без дополнительной изоляции. Проект предназначен для локального запуска (на личном ПК) или на выделенном VPS.

## ⚙️ Установка в 2 клика (Developer Experience)

Забудьте про `pip install`, конфликты версий C++ компиляторов и настройку баз данных. Весь проект упакован в Docker и разворачивается автоматически.

**Требования:** Установленный Docker и Python 3.11+ (только для запуска стартового скрипта на хосте).

1. **Склонируйте репозиторий:**

   ```bash

   git clone https://github.com/your-username/AAF_release.git

   cd AAF_release

   ```

2. **Запустите скрипт инициализации:**

   Скрипт сам создаст конфиги, скачает нужные NLP-модели на ваш диск (чтобы не качать их каждый раз в Докере), настроит пути песочницы и авторизует Telegram-сессию.

   ```bash

   python onboard.py

   ```

3. **Запустите систему:**

   ```bash

   docker-compose up -d --build

   ```

  
Всё! Ваш агент проснется, подключится к Telegram и начнет проактивно анализировать систему.

  
## 🧠 Настройка личности (Personality)

Вы можете превратить агента в кого угодно: от сухого корпоративного аналитика до саркастичного ИИ из видеоигр.

После запуска `onboard.py`, в корне проекта появится папка `config/personality/`. В ней лежат 3 файла:

*   `SOUL.md` - Ядро личности (кто он, чего хочет, как относится к миру).
*   `COMMUNICATION_STYLE.md` - Правила речи (использовать ли эмодзи, как обращаться к людям).
*   `EXAMPLES_OF_STYLE.md` - Примеры диалогов для настройки Few-Shot промптинга.

Просто измените текст в этих файлах, сохраните их, и при следующем пробуждении агент будет вести себя по-новому. Код трогать не нужно.

## 🛡 WatchDog & Self-Healing

Система обладает функцией самовосстановления. Если какой-либо модуль (например, подключение к БД или парсер) падает с ошибкой:

1. Модуль отправляет событие `SYSTEM_MODULE_ERROR` в шину.
2. Агент просыпается с максимальным приоритетом.
3. Получает Traceback ошибки.
4. Используя инструменты чтения локальных файлов, анализирует собственный исходный код.
5. Записывает отчет об ошибке или пытается перезапустить процесс.

## 🛠 Tech Stack

*   **Core:** Python 3.11+, Asyncio, Pydantic, Docker
*   **Memory:** PostgreSQL (SQLAlchemy + JSONB), ChromaDB (Vector), KuzuDB (Graph)
*   **Sensors:** Telethon (MTProto), psutil
*   **LLM Engine:** установка любых LLM моделей, начиная от Claude Opus 4.6 заканчивая GPT-5.4 и Gemini 3.1 Pro
